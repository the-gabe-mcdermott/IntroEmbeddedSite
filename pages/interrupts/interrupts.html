Interrupts

There are two ways software can check for events or input. They are Polling Driven Code and Interrupt Driven Code.

Polling Driven Code

Polling code is code that regularly reads the status of whatever event it is waiting for, usually within an endless loop.

While this is fine for simple applications, it begins to become insufficient when multiple inputs need to be handled at the same time.

The other major downside is the limited speed of response polling can provide. If a button is pressed, the software will not notice the event until the button read portion of the loop comes around again. In a complex program, this could be hundreds of cycles later, which could generate a noticeable delay.

Interrupt Driven Code

Interrupts are special events that trigger the processor to stop running whatever code it is currently running, and to briefly run a special function called an Interrupt Service Routine (or ISR), instead. When the ISR is finished, the processor resumes running the code it was running before.

Interrupt triggers can be set up to be tied to various kinds of events. Change of logic levels on an input pin, Timer overflow, completion of a serial transmission, and many more.
